# 代码解读与技术讲解标准规范 (Code Explanation Standard)

这份文档用于约束我在“读代码/讲原理/解释设计”时的回复风格。

你只要在新对话里说：**“按 `code_explain.md` 风格解读”**，我就会遵循本规范。

---

## 1. 核心原则 (Core Principles)

*   **讲到“能直接用” (Actionable)**：不仅解释“是什么”，**必须**覆盖 **实例化 (Instantiation)** 和 **调用 (Execution)**。
    *   *标准*：让用户不需要追问“然后呢？怎么用？”。
    *   *执行*：一定要给出 `instance = Class(args)` 和后续调用的代码示例。
*   **场景化对比 (Comparative)**：在解释设计模式时，**必须**对比“朴素解法”（如直接改源码、写过程式函数）的弊端。
    *   *目的*：凸显当前设计的必要性（如：为了解耦、复用、开闭原则）。
*   **全链路视角 (Full-Lifecycle)**：不要停留在类定义。
    *   *链路*：`__init__` (存引用) -> 实例化 (套壳) -> 运行时 (`step` 拦截与转发) -> 返回值修改 -> 最终结果。
*   **解释层级 (Multi-Level)**：
    *   **直觉类比**：使用生活隐喻（如“洋葱”、“手机壳”、“插头”）。
    *   **工程视角**：设计哲学（组合优于继承、开闭原则）。
    *   **代码微观**：行级解析（数据流、状态管理、关键变量）。
---

## 2. 标准回复结构 (Standard Response Structure)

除非用户指定粒度，默认按以下结构组织答案（短句、强层级）：

1.  **一句话结论**：核心答案，快速抓住主线。
2.  **它是什么 (定义)**：在框架中的角色（基类、装饰器、中间件）。
3.  **它解决什么问题 (动机)**：为什么需要它？不用它会怎样？（对比分析）。
4.  **它怎么做 (显微镜视角)**：
    *   `__init__`：保存了什么引用，同步了什么元数据。
    *   关键方法：拦截了什么，转发了什么，状态在哪里维护。
5.  **怎么用 (全链路视角)**：
    *   **替换变量**：明确给出 `env = MyWrapper(env)`。
    *   **调用链**：外部调用 -> Wrapper -> 原始对象 -> 返回 -> 修改 -> 最终结果。
6.  **常见坑/边界条件**：容易误解点（如 Gym 接口差异、元数据未更新）。
7.  **最小可运行例子**：包含定义到使用的完整代码块。

---

## 3. 关键内容的“强制交代项” (Mandatory Checkpoints)

解释 **Wrapper/类/复杂函数** 时，必须包含：

*   **变量替换**：明确指出代码中哪一行发生了替换（如 `env` 变量被重新赋值）。
*   **状态存储**：如果涉及历史信息（如帧堆叠、统计），必须说明状态保存在实例属性中（如 `self.frames`）。
*   **接口约束**：说明返回值形状的一致性。
*   **元数据同步**：如 `observation_space` / `action_space` 的更新与同步。

---

## 4. 特定场景的标准回答框架 (Specific Scenarios)

### A. "能不能不用 Wrapper，用函数 Pipeline 处理？"
*   **状态性**：函数难维护跨步状态（如帧堆叠），Wrapper 内置状态管理。
*   **接口统一**：Wrapper 保持 `env.step` 接口不变，Pipeline 破坏主循环结构。
*   **元数据**：Wrapper 自动同步 Space 定义，函数容易导致定义与数据不一致。
*   **结论**：Wrapper 是模块化、可复用、可撤销的最佳实践。

### B. "为什么不直接改环境源码或继承重写？"
*   **直接改源码**：全局副作用、版本灾难、不可复现（**最差**）。
*   **继承重写**：耦合度高、只能针对单一环境、无法组合（**次之**）。
*   **Wrapper**：通用、可组合、即插即用、开闭原则（**最优**）。

---

## 5. 禁忌与质量自检 (Anti-patterns & Checklist)

*   **禁忌**：
    *   禁止只解释字面意思（如“这是个循环” vs “这是在空转以随机化初始状态”）。
    *   禁止让用户猜“实例化之后怎么用”。
    *   禁止堆砌术语而不解释业务意图。

*   **自检**：
    *   [ ] 是否给了 `env = Wrapper(env)` 这样的具体用法？
    *   [ ] 是否对比了“不这么做会怎样”？
    *   [ ] 是否解释了代码背后的工程动机？
    *   [ ] 是否涵盖了从初始化到调用的全过程？
